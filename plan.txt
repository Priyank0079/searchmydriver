Subscription Feature Implementation Plan
The goal is to complete the subscription flow so that users can purchase full-time driver subscriptions, and admins can manage these requests by zone and assign drivers.

Current State Analysis
Backend Models: SubscriptionPlan and UserSubscription already exist and support driver assignment.
Backend Pricing Service: Supports CRUD for plans, fetching active plans, estimating fare with discounts, and admin assigning/releasing drivers.
Frontend Admin: ManageSubscriptions.jsx exists and manages SubscriptionPlans (pricing).
Frontend User: SubscriptionsPage.jsx exists and displays active plans but has a "Subscribe — coming soon" placeholder button and no payment integration.
Proposed Changes
1. Database Model Updates
[MODIFY] 
userSubscription.model.js
Add zoneId: { type: mongoose.Schema.Types.ObjectId, ref: 'Zone', required: true, index: true } to associate the subscription request with a specific operational zone, matching the outstation pattern.
Ensure the status defaults to PENDING_PAYMENT initially, moving to ACTIVE upon successful Razorpay verification.
2. Backend Services & Controllers (Payment Flow)
[MODIFY] 
pricing.service.js
Add createSubscriptionPurchaseOrderService(userId, planId, zoneId):
Validates the user and plan.
Creates a UserSubscription record in PENDING_PAYMENT status.
Generates a Razorpay order for the plan's price.
Returns the Razorpay key and order details to the frontend.
Add verifySubscriptionPaymentService(userId, { razorpayOrderId, razorpayPaymentId, razorpaySignature }):
Validates the Razorpay signature.
Updates the UserSubscription to status: ACTIVE, sets paidAt, and assignmentStatus: PENDING.
Modify listUserSubscriptionsService to accept a zoneId filter for the admin panel.
[MODIFY] 
pricing.controller.js
Add user-facing controllers: purchaseSubscription and verifySubscriptionPayment.
[MODIFY] 
user.routes.js
Wire up POST /subscriptions/purchase and POST /subscriptions/verify-payment to the new controllers.
3. Frontend User App (Purchase Flow)
[MODIFY] 
SubscriptionsPage.jsx
Update the "coming soon" button to trigger the purchase flow.
Call the /subscriptions/purchase API to get the Razorpay order.
Open the Razorpay checkout overlay.
On success, call /subscriptions/verify-payment.
Navigate the user to a "My Subscription" view or show a success state.
Prompt the user to select their zoneId before purchasing (or infer it from their saved location/current location if possible).
4. Frontend Admin Panel (Zone-wise Assignment)
[NEW] 
ManageUserSubscriptions.jsx
Create a new admin page dedicated to handling incoming UserSubscription requests.
Add a Zone selector dropdown to filter requests by zone (similar to outstation).
Display a list or kanban board of subscriptions categorized by assignmentStatus (PENDING, ASSIGNED).
Provide an interface for Admins to:
Assign a driver: Select a driver from the pool and assign them to the subscription.
Reassign a driver: Release the current driver (due to absence) and assign a new one, using the existing /subscriptions/users/:id/release and assign endpoints.
[MODIFY] Admin Sidebar/Navigation
Add a navigation link to the new ManageUserSubscriptions.jsx page (e.g., "Subscription Requests").
Open Questions
IMPORTANT

Zone Selection: When a user purchases a subscription on the app, how should we determine their zoneId? Should we use their active zone, ask them to select a city, or use their primary saved home location?
Razorpay Plan Price: Are there any additional fees (GST, service charge) added to the base subscription price at checkout, or is the price field in SubscriptionPlan fully inclusive?
